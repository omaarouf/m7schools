## Partie Théorique (16 pts)

### Q1 — LVM signifie : (2 pts)

- A. Lightweight Volume Manager
- B. **Logical Volume Manager**
- C. Linux Virtual Memory
- D. Linux VOD Manager

<details>
<summary>Voir la réponse</summary>

**Réponse : B — Logical Volume Manager**

LVM est une couche d'abstraction entre les disques physiques et le système de fichiers permettant de créer des volumes logiques flexibles et redimensionnables.

</details>

---

### Q2 — Avantage principal de LVM par rapport aux partitions traditionnelles : (2 pts)

- A. LVM permet la création de partitions plus petites
- B. LVM permet la création de partitions plus grandes
- C. **LVM offre la flexibilité de redimensionner les volumes logiques à chaud**
- D. LVM garantit une meilleure performance des disques durs

<details>
<summary>Voir la réponse</summary>

**Réponse : C — LVM offre la flexibilité de redimensionner les volumes logiques à chaud**

Contrairement aux partitions classiques, les volumes LVM peuvent être étendus ou réduits sans démonter le système ni redémarrer le serveur.

</details>

### Q3 — Avantage associé au RAID 0 : (2 pts)

- A. Redondance des données pour la tolérance aux pannes
- B. **Amélioration de la performance en termes de vitesse de lecture/écriture**
- C. Protection des données contre la défaillance d'un disque

<details>
<summary>Voir la réponse</summary>

**Réponse : B — Amélioration de la performance**

Le RAID 0 (striping) répartit les données sur plusieurs disques simultanément, augmentant la vitesse de lecture/écriture. Il n'offre **aucune redondance** — si un disque tombe en panne, toutes les données sont perdues.

</details>

---

### Q4 — Comment le RAID 1 gère-t-il la redondance : (2 pts)

- A. **Les données sont répliquées sur plusieurs disques**
- B. Les données sont divisées en blocs et réparties sur plusieurs disques
- C. Les données sont compressées pour économiser de l'espace

<details>
<summary>Voir la réponse</summary>

**Réponse : A — Les données sont répliquées sur plusieurs disques**

Le RAID 1 (mirroring) écrit les données identiquement sur deux disques ou plus. Si un disque tombe en panne, les données sont toujours disponibles sur le disque miroir.

</details>

### Q5 — Rôle du service NFS : (2 pts)

<details>
<summary>Voir la réponse</summary>

**NFS** (Network File System) est un protocole de partage de fichiers réseau qui permet à des machines Linux/Unix d'**accéder à des répertoires distants** comme s'ils étaient locaux.

- Le serveur NFS exporte des répertoires via `/etc/exports`
- Le client monte ces répertoires avec `mount` ou via `/etc/fstab`
- Transparent pour les applications — elles ne voient pas la différence entre un fichier local et distant

```bash
# Côté serveur
/srv/partage    192.168.2.0/24(rw,sync)

# Côté client
sudo mount 192.168.2.10:/srv/partage /mnt/nfs
```

</details>

### Q6 — Rôle de la commande `ssh-keygen` : (2 pts)

<details>
<summary>Voir la réponse</summary>

`ssh-keygen` génère une **paire de clés cryptographiques** pour l'authentification SSH :

- **Clé privée** (`~/.ssh/id_rsa`) — reste sur la machine locale, ne jamais partager
- **Clé publique** (`~/.ssh/id_rsa.pub`) — copiée sur le serveur distant dans `~/.ssh/authorized_keys`

```bash
ssh-keygen -t rsa -b 2048    # Génère une paire de clés RSA 2048 bits
ssh-copy-id user@serveur     # Copie la clé publique vers le serveur
```

Permet de se connecter **sans mot de passe** tout en étant plus sécurisé.

</details>

### Q7 — Directive `anon_upload_enable NO` dans vsftpd : (1 pt)

<details>
<summary>Voir la réponse</summary>

Cette directive **interdit l'upload de fichiers** aux utilisateurs anonymes sur le serveur FTP.

- `NO` (valeur par défaut) — les anonymes ne peuvent que télécharger (download), pas envoyer de fichiers
- `YES` — autorise les anonymes à uploader (dangereux en production)

</details>

---

### Q8 — Directive `PermitRootLogin YES` dans `sshd_config` : (1 pt)

<details>
<summary>Voir la réponse</summary>

Cette directive **autorise la connexion SSH directe avec le compte root**.

:::danger Risque de sécurité
`PermitRootLogin YES` est fortement déconseillé en production — si un attaquant compromet la connexion SSH, il obtient immédiatement les droits root. La bonne pratique est de mettre `PermitRootLogin no` et d'utiliser `sudo`.
:::

</details>

### Q9 — Fichier de configuration du service DNS : (1 pt)

<details>
<summary>Voir la réponse</summary>

```
/etc/named.conf          # Fedora / RHEL — fichier principal BIND
/etc/bind/named.conf     # Ubuntu/Debian — fichier principal BIND
```

Fichiers associés :
- `/var/named/` — répertoire des fichiers de zones (Fedora)
- `/etc/bind/` — répertoire des fichiers de zones (Ubuntu)

### Q10 — Rôle des enregistrements DNS `AAAA` et `MX` : (1 pt)

<details>
<summary>Voir la réponse</summary>

| Type | Rôle | Exemple |
|---|---|---|
| **AAAA** | Associe un nom d'hôte à une adresse **IPv6** (équivalent de A pour IPv6) | `srv1 IN AAAA 2001:db8::1` |

**Q5. Configurer le serveur DHCP selon les paramètres donnés. (4 pts)**

Paramètres : sous-réseau `192.168.2.0/24` · plage `192.168.2.30` à `192.168.2.150` · passerelle `192.168.2.1` · DNS `192.168.2.2` · domaine `examff.ma`

<details>
<summary>Voir la réponse</summary>

```bash
# /etc/dhcp/dhcpd.conf

default-lease-time 600;
max-lease-time 7200;
authoritative;

subnet 192.168.2.0 netmask 255.255.255.0 {
    range 192.168.2.30 192.168.2.150;
    option routers 192.168.2.1;
    option domain-name-servers 192.168.2.2;
    option domain-name "examff.ma";
}
```

:::warning Erreur dans le sujet
La plage indiquée `192.168.2.30` à `192.168.1.150` contient une incohérence (réseau différent). La plage correcte est `192.168.2.30` à `192.168.2.150`.
:::

Démarrer le service :

```bash
# Fedora
sudo systemctl restart dhcpd
sudo systemctl enable dhcpd

# Ubuntu
sudo systemctl restart isc-dhcp-server
sudo systemctl enable isc-dhcp-server
```

</details>

**Q6. Réservation de `192.168.2.30` pour `SRV-WEB` (MAC : `26-77-03-F2-E2-C2`). (3 pts)**

<details>
<summary>Voir la réponse</summary>

Dans `/etc/dhcp/dhcpd.conf`, ajouter après le bloc subnet :

```bash
host SRV-WEB {
    hardware ethernet 26:77:03:F2:E2:C2;
    fixed-address 192.168.2.30;
}
```

:::info Format de l'adresse MAC
Les tirets `-` du sujet se remplacent par des deux-points `:` dans la configuration DHCP Linux.
:::

</details>

**Q8. Créer un VirtualHost pour `www.regional.ma` — dossier `/var/www/site1` — port `80`. (3 pts)**

<details>
<summary>Voir la réponse</summary>

Créer le fichier `/etc/httpd/conf.d/regional.conf` (Fedora) ou `/etc/apache2/sites-available/regional.conf` (Ubuntu) :

```apache
<VirtualHost *:80>
    ServerName www.regional.ma
    DocumentRoot /var/www/site1
    ServerAdmin admin@regional.ma
    ErrorLog /var/log/httpd/regional-error.log
    CustomLog /var/log/httpd/regional-access.log combined
</VirtualHost>
```

Créer le répertoire et activer le site (Ubuntu) :

```bash
sudo mkdir -p /var/www/site1
echo "<h1>www.regional.ma</h1>" | sudo tee /var/www/site1/index.html

# Ubuntu uniquement
sudo a2ensite regional.conf
sudo systemctl reload apache2
```

</details>
