### Question 4 — Expliquer la configuration DHCP (2 pts)

```
Host IDSR {
    Hardware Ethernet 00:bc:a0:77:e0:12;
    fixed-address 192.168.10.3;
}
```

<details>
<summary>Voir la réponse</summary>

Il s'agit d'une **réservation DHCP statique** :

- `host IDSR` — nom logique de la réservation
- `hardware ethernet 00:bc:a0:77:e0:12` — adresse MAC de la carte réseau du client
- `fixed-address 192.168.10.3` — adresse IP fixe toujours attribuée à ce client

Chaque fois que la machine avec cette adresse MAC demandera une IP, le serveur DHCP lui attribuera toujours `192.168.10.3`.

</details>

### Question 6 — Rôle de chmod (1 pt)

Quel est le rôle de la commande `chmod` ?

<details>
<summary>Voir la réponse</summary>

`chmod` (change mode) permet de modifier les **permissions d'accès** d'un fichier ou répertoire pour les 3 catégories : propriétaire (u), groupe (g) et autres (o).

```bash
chmod 755 /var/www/html       # rwxr-xr-x
chmod u+x script.sh           # ajouter l'exécution au propriétaire
chmod -R 644 /etc/apache2/    # récursif
```

| Valeur | Permission | Signification |
|---|---|---|
| 4 | r | Lecture |
| 2 | w | Écriture |
| 1 | x | Exécution |

</details>

**6. Expliquer les lignes en gras du fichier de zone. (2 pts)**

```
$TTL 86400
@ IN SOA DNS1.id.local. root.ntic.local. (
    2023040100; Serial
    3600       ; Refresh
    1800       ; Retry
    604800     ; Expire
    86400      ; Minimum TTL
)
```

Les lignes à expliquer : `2023040100; Serial` · `1800 ; Retry` · `604800 ; Expire`

<details>
<summary>Voir la réponse</summary>

- **`2023040100 ; Serial`** — Numéro de série de la zone (format AAAAMMJJNN). Doit être incrémenté à chaque modification pour que les serveurs secondaires détectent la mise à jour.
- **`1800 ; Retry`** — Délai (30 minutes) avant qu'un serveur secondaire réessaie de contacter le primaire après un échec de rafraîchissement.
- **`604800 ; Expire`** — Durée (7 jours) après laquelle les données de zone sont invalides sur le secondaire s'il ne joint pas le primaire.

</details>

---

**7. Compléter le fichier de zone directe avec les enregistrements RR. (3 pts)**

Cahier des charges :
- Deux clients : `clt1` (192.168.10.5) et `clt2` (192.168.10.6)
- Serveur DNS principal : DNS1
- Noms canoniques pour DNS1 : `www`, `ldap`
- Serveur de messagerie : serveur2
- Serveur DHCP : serveur3

<details>
<summary>Voir la réponse</summary>

```bash
$TTL 86400
@   IN  SOA  DNS1.id.local.  root.ntic.local. (
            2023040100  ; Serial
            3600        ; Refresh
            1800        ; Retry
            604800      ; Expire
            86400 )     ; Minimum TTL

; Serveur de noms
@       IN  NS      DNS1.id.local.

; Enregistrements A
DNS1    IN  A       192.168.10.1
serveur2 IN A       192.168.10.2
serveur3 IN A       192.168.10.3
clt1    IN  A       192.168.10.5
clt2    IN  A       192.168.10.6

; Alias CNAME
www     IN  CNAME   DNS1.id.local.
ldap    IN  CNAME   DNS1.id.local.

; Enregistrement MX
@       IN  MX  10  serveur2.id.local.
```

</details>

**16. Compléter le fichier de configuration du VirtualHost pour `www.ntic.ma`. (4 pts)**

```
<VirtualHost * : 1>
    DocumentRoot 2
    3 www.ntic.ma
<4>
```

<details>
<summary>Voir la réponse</summary>

| N° | Valeur | Rôle |
|---|---|---|
| 1 | `80` | Port d'écoute HTTP |
| 2 | `/var/www/ntic` | Répertoire racine du site |

```apache
    ServerName www.ntic.ma
    ServerAdmin admin@ntic.ma
    ErrorLog /var/log/httpd/ntic-error.log
    CustomLog /var/log/httpd/ntic-access.log combined
</VirtualHost>
```

### Exercice 4 — Accès distant SSH (2 pts)

On veut accéder via SSH depuis serveur2 au serveur1 avec une authentification RSA.

Commande exécutée sur serveur2 : `ssh-keygen -t rsa -b 2048`

---

**17a. Expliquer à quoi sert la commande exécutée. (1 pt)**

<details>
<summary>Voir la réponse</summary>

Génère une **paire de clés RSA** pour l'authentification SSH :

- `-t rsa` — algorithme RSA
- `-b 2048` — longueur de la clé : 2048 bits

Crée deux fichiers dans `~/.ssh/` :
- `id_rsa` — **clé privée** (rester sur serveur2, ne jamais partager)
- `id_rsa.pub` — **clé publique** (à copier sur serveur1 dans `~/.ssh/authorized_keys`)

Permet de se connecter à serveur1 depuis serveur2 **sans mot de passe**.

</details>

---

**17b. Que contient le fichier `id_rsa` ? (1 pt)**

<details>
<summary>Voir la réponse</summary>

Le fichier `id_rsa` contient la **clé privée RSA** de l'utilisateur. Il est protégé par une passphrase optionnelle et doit avoir les permissions `600` (lisible uniquement par le propriétaire). Il ne doit jamais être partagé.

</details>

### Exercice 5 — Partage SAMBA (3 pts)

Le serveur2 est aussi un serveur SAMBA. On veut partager le répertoire `/exam` en lecture avec `user_share1` et `user_share2`.

---

**18a. Créer le répertoire `/exam` et les deux utilisateurs SAMBA avec mot de passe `samba`. (1 pt)**

<details>
<summary>Voir la réponse</summary>

```bash
# Créer le répertoire
sudo mkdir -p /exam
sudo chmod 755 /exam

# Créer les utilisateurs système
sudo useradd -M -s /sbin/nologin user_share1
sudo useradd -M -s /sbin/nologin user_share2

# Créer les utilisateurs SAMBA
sudo smbpasswd -a user_share1
sudo smbpasswd -a user_share2

# Activer les comptes SAMBA
sudo smbpasswd -e user_share1
sudo smbpasswd -e user_share2
```

</details>

---

**18b. Éditer `/etc/samba/smb.conf` et ajouter la section du partage voulu. (2 pts)**

<details>
<summary>Voir la réponse</summary>

```ini
[exam]
    comment = Partage répertoire exam
    path = /exam
    browseable = yes
    read only = yes
    valid users = user_share1 user_share2
    create mask = 0644
    directory mask = 0755
```

Puis vérifier et redémarrer :

```bash
sudo testparm

# Fedora / RHEL
sudo systemctl restart smb nmb

# Ubuntu
sudo systemctl restart smbd nmbd
```

</details>
