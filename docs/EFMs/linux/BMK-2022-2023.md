### Q2 — Contenu du fichier `/etc/resolv.conf` (1 pt)

<details>
<summary>Voir la réponse</summary>

Le fichier `/etc/resolv.conf` contient la **configuration du résolveur DNS client** du système :

```bash
# Exemple de contenu
search efm.reg
nameserver 192.168.1.10
nameserver 8.8.8.8
```

- `search` — suffixe DNS utilisé pour compléter les noms courts
- `nameserver` — adresse IP du ou des serveurs DNS à interroger (jusqu'à 3)

</details>

### Q3 — Les quatre requêtes du processus de bail DHCP (1 pt)

<details>
<summary>Voir la réponse</summary>

Le processus **DORA** :

| Étape | Type | Description |
|---|---|---|
| 1 | **Discover** | Le client diffuse une demande d'adresse IP (broadcast) |
| 2 | **Offer** | Le serveur propose une adresse IP disponible |
| 3 | **Request** | Le client accepte l'offre et la demande officiellement |
| 4 | **Acknowledge** | Le serveur confirme l'attribution du bail |

</details>

### Q4 — Directive `option routers 192.168.10.1` dans dhcpd.conf (1 pt)

<details>
<summary>Voir la réponse</summary>

Cette directive indique aux clients DHCP l'**adresse IP de la passerelle par défaut** (default gateway) à utiliser pour communiquer avec des réseaux extérieurs. Ici, `192.168.10.1` sera configuré comme passerelle sur chaque client qui obtient une adresse via ce serveur DHCP.

</details>

---

### Q5 — Rôle du bloc de réservation DHCP (1 pt)

```
host server01 {
    hardware ethernet 08:00:07:26:00:5;
    fixed-address 192.168.10.10;
}
```

<details>
<summary>Voir la réponse</summary>

Il s'agit d'une **réservation DHCP statique** pour la machine `server01` :

- `hardware ethernet 08:00:07:26:00:5` — identifie le client par son adresse MAC
- `fixed-address 192.168.10.10` — adresse IP fixe qui lui sera toujours attribuée

Chaque fois que cette machine demandera une IP, le serveur DHCP lui attribuera toujours `192.168.10.10`.

</details>

### Q6 — Rôle des commandes ACL (1 pt)

<details>
<summary>Voir la réponse</summary>

**`setfacl -m u:yahya:rwx /mnt/partage`**
Ajoute ou modifie une ACL pour donner à l'utilisateur `yahya` les droits **lecture, écriture et exécution** sur `/mnt/partage`, sans modifier le groupe propriétaire ni les permissions classiques.

**`getfacl /mnt/partage`**
Affiche toutes les ACL (Access Control Lists) appliquées sur `/mnt/partage` — permissions classiques et permissions ACL étendues pour chaque utilisateur/groupe concerné.

</details>

### Q8 — Autoriser le trafic Web dans le pare-feu (1 pt)

<details>
<summary>Voir la réponse</summary>

**Avec firewalld (Fedora/RHEL) :**

```bash
sudo firewall-cmd --add-service=http --permanent
sudo firewall-cmd --add-service=https --permanent
sudo firewall-cmd --reload
```

**Avec iptables :**

```bash
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
sudo service iptables save
```

**Avec ufw (Ubuntu) :**

```bash
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
```

</details>

### Q10 — Directives `PermitRootLogin no` et `PasswordAuthentication no` dans `sshd_config` (1 pt)

<details>
<summary>Voir la réponse</summary>

**`PermitRootLogin no`**
Interdit la connexion SSH directe avec le compte root. Les administrateurs doivent se connecter avec un compte utilisateur standard puis utiliser `sudo`. Recommandé en production.

**`PasswordAuthentication no`**
Interdit l'authentification SSH par mot de passe. Seule l'authentification par **clé publique** est autorisée. Beaucoup plus sécurisé — protège contre les attaques par force brute.

:::tip
Ces deux directives combinées représentent la configuration SSH la plus sécurisée — accès uniquement via clé RSA, root inaccessible directement.
:::

</details>

**Q5. Fichier de zone directe — enregistrements A, NS et CNAME pour `DNSWEB`. (3 pts)**

<details>
<summary>Voir la réponse</summary>

```bash
$TTL 86400
@   IN  SOA  DNSWEB.efm.reg.  admin.efm.reg. (
            2023021101  ; Serial
            3600        ; Refresh
            1800        ; Retry
            604800      ; Expire
            86400 )     ; Minimum TTL

; Enregistrement NS
@       IN  NS      DNSWEB.efm.reg.

; Enregistrement A
DNSWEB  IN  A       192.168.1.10

; Enregistrement CNAME — alias www
www     IN  CNAME   DNSWEB.efm.reg.
```

</details>

**Q6. Fichier de zone inversée — enregistrement PTR pour `DNSWEB`. (3 pts)**

<details>
<summary>Voir la réponse</summary>

```bash
$TTL 86400
@   IN  SOA  DNSWEB.efm.reg.  admin.efm.reg. (
            2023021101  ; Serial
            3600        ; Refresh
            1800        ; Retry
            604800      ; Expire
            86400 )     ; Minimum TTL

@       IN  NS      DNSWEB.efm.reg.

; Enregistrement PTR — résolution inverse
10      IN  PTR     DNSWEB.efm.reg.
```

:::info
`10` correspond au dernier octet de l'IP `192.168.1.10`. La résolution inverse de `192.168.1.10` retournera `DNSWEB.efm.reg`.
:::

</details>

**Q7. Changer le groupe propriétaire des fichiers de zone en `named`. (3 pts)**

<details>
<summary>Voir la réponse</summary>

```bash
sudo chown named:named /var/named/efm.reg.dir
sudo chown named:named /var/named/efm.reg.inv

# Ou les deux en une commande
sudo chown named:named /var/named/efm.reg.*

# Vérifier
ls -l /var/named/
```

:::info
Le groupe `named` est le groupe système utilisé par le processus BIND pour accéder aux fichiers de zone. Sans cette étape, BIND ne peut pas lire les fichiers de zone et le service refusera de démarrer.
:::

</details>

**Q10. Configurer deux VirtualHosts sur le serveur web. (4 pts)**

- **Site 1 :** dossier `/var/www/v1/site1` · nom `site1.efm.reg`
- **Site 2 :** dossier `/var/www/v1/site2` · nom `site2.efm.reg`

<details>
<summary>Voir la réponse</summary>

Créer le fichier `/etc/httpd/conf.d/efm.conf` (Fedora) :

```apache
# Premier site
<VirtualHost *:80>
    ServerName site1.efm.reg
    DocumentRoot /var/www/v1/site1
    ServerAdmin admin@efm.reg
    ErrorLog /var/log/httpd/site1-error.log
    CustomLog /var/log/httpd/site1-access.log combined
</VirtualHost>

# Deuxième site
<VirtualHost *:80>
    ServerName site2.efm.reg
    DocumentRoot /var/www/v1/site2
    ServerAdmin admin@efm.reg
    ErrorLog /var/log/httpd/site2-error.log
    CustomLog /var/log/httpd/site2-access.log combined
</VirtualHost>
```

Créer les répertoires et fichiers index :

```bash
sudo mkdir -p /var/www/v1/site1
sudo mkdir -p /var/www/v1/site2
echo "<h1>Site1 - efm.reg</h1>" | sudo tee /var/www/v1/site1/index.html
echo "<h1>Site2 - efm.reg</h1>" | sudo tee /var/www/v1/site2/index.html

# Vérifier la syntaxe
sudo apachectl configtest
```

</details>
